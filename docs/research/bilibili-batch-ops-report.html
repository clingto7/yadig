Bilibili 批量收藏、关注、订阅管理调研报告

调研日期：2026-06-10；项目上下文：yadig 已有 Bilibili 登录、视频信息和音频提取能力。

本轮目标是判断能否对 Bilibili 网站上的收藏、关注、订阅做批量分类处理，并给出可落地的实现路线。结论是：收藏夹和关注分组具备较清晰的 Web 端接口路径，适合做成桌面端“预览后执行”的批量工具；订阅需要先拆成不同业务对象，不能作为一个统一接口处理。

收藏夹：可优先实现 关注分组：可优先实现 订阅：需拆对象 接口非官方稳定承诺

一、执行摘要

最可行路线

在现有 Tauri 应用中扩展 Bilibili 账号管理模块：扫码登录或完整 Cookie 导入后，同步收藏夹、关注列表、关注分组，生成批量操作计划，由用户确认后限速执行。

第一期范围

收藏夹：批量创建、移动、复制、删除内容、清理失效内容。关注：读取关注列表和分组，批量创建分组、复制或移动关注用户到分组。

主要风险

这些接口来自 Web 端逆向资料，不是官方开放平台承诺的稳定 API。需要限速、重试、操作日志、幂等检查、失败回滚提示，并避免自动化高频行为。

建议决策：不要先做浏览器脚本或全自动分类机器人。更稳妥的 MVP 是“读取账号数据 → 本地打标签/规则筛选 → 展示差异 → 用户确认 → 小批量执行 → 记录可追溯日志”。

二、项目现状

当前仓库已有 Bilibili 基础能力，适合作为批量管理工具的底座：

  • Rust 侧已有 BiliAuth 与 BiliSession，会话字段包含 sessdata、bili_jct、dede_user_id。
  • 二维码登录流程会从回调 URL 提取 SESSDATA、bili_jct、DedeUserID，因此具备写接口所需的 CSRF token 来源。
  • 仅粘贴 SESSDATA 的 Cookie 登录路径目前只保存 SESSDATA，bili_jct 为空；这对读取接口通常够用，但对收藏夹修改、关注分组修改、追番/取消追番这类 POST 写接口不够。
  • 现有 HTTP 客户端已经有 Referer 和 Cookie 注入模式，可以复用，但写操作需要统一封装 csrf=bili_jct、错误码处理和限速。

三、功能可行性

对象 可批量处理的动作 接口成熟度 一期建议
用户创建的收藏夹 读取收藏夹列表、读取内容、创建/编辑/删除收藏夹、批量复制/移动/删除收藏内容、清理失效内容。 较高。公开逆向文档覆盖读取和写入接口，参数结构清楚。 做 作为第一期主功能。
关注和关注分组 读取关注列表、搜索关注、读取分组、创建/改名/删除分组、批量复制/移动关注用户到分组。 较高。分组管理接口完整，但关注列表存在分页和可见性限制。 做 作为第一期主功能。
追番/追剧 追番、取消追番；可读取番剧相关状态需要另查 PGC 接口。 中等。跟随/取消接口明确，但“分类处理”不是同一层语义。 暂缓 第二期做本地标签，不急着改远端。
订阅合集、频道、列表、UP 主动态等 概念分散，接口域不同，不存在一个统一“订阅”对象。 待验证。需要先定义具体对象和用户想要的分类结果。 拆分调研 先不要混进 MVP。

四、关键接口线索

以下端点来自社区维护的 Bilibili Web/API 收集文档，属于非官方接口线索；正式实现前应使用小号或测试收藏夹实测。

收藏夹读取

用途 端点 关键参数
获取用户创建的所有收藏夹 GET /x/v3/fav/folder/created/list-all up_mid、可携带 SESSDATA 查看自己的私密收藏夹。
获取收藏夹内容明细 GET /x/v3/fav/resource/list media_id、pn、ps、order、platform=web。
获取收藏夹全部内容 id GET /x/v3/fav/resource/ids media_id、platform=web。
批量获取资源详情 GET /x/v3/fav/resource/infos resources={id}:{type}，类型常见有视频 2、音频 12、合集 21。

收藏夹写入

用途 端点 关键参数
创建收藏夹 POST /x/v3/fav/folder/add title、intro、privacy、csrf。
批量复制内容 POST /x/v3/fav/resource/copy src_media_id、tar_media_id、mid、resources、csrf。
批量移动内容 POST /x/v3/fav/resource/move src_media_id、tar_media_id、mid、resources、csrf。
批量删除收藏内容 POST /x/v3/fav/resource/batch-del resources、media_id、platform=web、csrf。
清理失效内容 POST /x/v3/fav/resource/clean media_id、csrf。

关注和分组

用途 端点 关键参数和限制
读取关注列表 GET /x/relation/followings vmid、pn、ps。登录可看当前用户全部；其他用户通常只能看前 100 个，且要求 Referer 合理、UA 不含明显脚本标识。
读取关注分组 GET /x/relation/tags Cookie 或 APP 鉴权。
创建/改名/删除分组 POST /x/relation/tag/create、/tag/update、/tag/del 分组名最长 16 字符；Cookie 方式需要 csrf。
添加、复制、移动关注用户到分组 POST /x/relation/tags/addUsers、/copyUsers、/moveUsers fids 为用户 mid 列表，tagids 或 beforeTagids/afterTagids 为分组 id 列表，Cookie 方式需要 csrf。

追番/取消追番

用途 端点 关键参数
追番 POST /pgc/web/follow/add season_id、csrf。
取消追番 POST /pgc/web/follow/del season_id、csrf。

五、可选实现方式对比

方式 优点 问题 结论
Tauri 原生客户端调用 Web 接口 复用现有登录和 Rust HTTP 客户端；可保存本地数据库、计划、日志；适合做预览、限速和错误恢复。 需要维护非官方接口兼容性；需要扩展完整 Cookie/CSRF 管理。 推荐 作为主路线。
浏览器扩展或用户脚本 天然使用浏览器登录态，少处理 Cookie。 权限、跨域、页面结构变化、分发和安全边界更麻烦；难做稳定批处理日志。 可作为临时验证 不建议产品化优先。
Playwright/浏览器自动化 无需逆向部分接口，可模拟用户操作。 慢、脆弱、容易触发风控；界面变化会直接破坏流程。 不推荐 只适合人工辅助。
官方开放平台 合规性和稳定性最好。 公开开放平台并未覆盖个人账号收藏夹、关注分组这类完整批量管理能力。 持续关注 不能支撑本需求主体。

六、推荐架构

数据流

登录态验证
  -> 同步远端收藏夹/关注/分组快照
  -> 本地规则或人工筛选生成操作计划
  -> 预览差异：将移动/复制/删除哪些对象
  -> 用户确认
  -> 按小批次限速执行
  -> 记录成功、失败、远端返回码和可重试项

后端模块建议

  • BiliAccountClient：统一注入 SESSDATA、bili_jct、Referer、User-Agent，封装 GET/POST 表单请求。
  • BiliFavoritesService：收藏夹同步、内容分页、批量移动/复制/删除。
  • BiliRelationService：关注列表同步、关注分组同步、分组写入操作。
  • BatchPlan：本地操作计划模型，保存 source、target、resource id、resource type、当前状态、目标状态。
  • BatchExecutor：限速、分批、重试、错误分类、日志落库。

前端功能建议

  • 账号页提示登录方式：推荐扫码登录；高级 Cookie 登录必须输入完整 Cookie 或至少 SESSDATA + bili_jct。
  • 收藏页提供表格筛选：收藏夹、UP 主、标题关键词、失效状态、发布时间、收藏时间。
  • 关注页提供分组矩阵：左侧关注用户列表，右侧当前分组和目标分组，支持多选移动/复制。
  • 执行前必须显示操作预览，尤其是删除和移动操作；默认关闭“删除远端内容”。
  • 执行后提供日志导出，便于人工核对。

七、分类策略设计

策略 适用对象 实现成本 说明
关键词规则 收藏夹内容、关注用户 低 按标题、简介、UP 主昵称、分区、标签命中规则，透明可解释，适合 MVP。
已有分组映射 关注分组 低 例如“游戏/音乐/技术/生活”映射到远端分组，先人工维护规则。
本地标签层 所有对象 中 不立即改远端，先在本地保存标签和建议；确认后再批量执行。
LLM 辅助分类 收藏内容和关注用户 中高 可基于标题、简介、UP 主信息生成建议，但必须作为“建议”，不能直接自动执行远端修改。

八、风险和边界

账号与合规风险：批量接口调用可能触发 Bilibili 风控，也可能违反平台对自动化、异常访问或账号安全的要求。实现时应以用户本人账号、低频、可确认、可撤销/可追溯为原则，并避免绕过验证码、风控或权限限制。
  • 接口稳定性：Web 端接口可能改名、改参数、加签名或调整返回码。
  • 写操作不可逆：删除收藏夹、删除收藏内容、取消追番等应默认不做，除非用户显式选择。
  • CSRF 要求：所有写接口都依赖 bili_jct 作为 csrf，仅保存 SESSDATA 不够。
  • 分页限制：关注列表对他人账号有可见性和数量限制；本需求应限定为“管理当前登录用户自己的数据”。
  • 速率控制：建议每批 10-20 个资源，批次之间等待 1-3 秒；遇到 -101、-111、-400、412 或验证码/风控提示时立即停止。

九、一期落地计划

  1. 补齐登录态：扫码登录作为默认；Cookie 登录改为完整 Cookie 导入，至少解析 SESSDATA、bili_jct、DedeUserID。
  2. 新增只读同步：实现收藏夹列表、收藏内容、关注列表、关注分组读取，并落本地快照。
  3. 新增操作计划：支持用户选择若干收藏内容或关注用户，生成移动/复制计划。
  4. 实现写入执行器：先支持收藏夹内容复制/移动、关注用户复制/移动分组；删除类操作放到后面。
  5. 加保护机制：预览确认、限速、失败暂停、操作日志、重复执行前检查当前远端状态。
  6. 用测试收藏夹和少量关注分组做端到端验证，记录每个接口的真实返回码。

十、调研结论

本需求可以做，但应先聚焦“收藏夹整理”和“关注分组整理”。这两类对象有清楚的远端结构和批量接口，和当前 yadig 的 Bilibili 登录/客户端基础也匹配。

“订阅”不是一个单一能力，需要在下一轮明确是追番/追剧、订阅合集、频道、UP 主动态，还是其他对象。追番/取消追番接口可作为独立能力，但不建议混入第一期批量分类 MVP。
